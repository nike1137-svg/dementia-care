# dementia-care — 위협 모델 (Threat Model) v1

- **작성일**: 2026-07-17
- **대상**: marcus-desktop(리눅스 데스크탑)을 Docker로 서버화해 외부 노출하는 개인 1차 프로젝트
- **목적**: "조심하기"가 아니라 **"실수해도 도달 못 하게"** 구조를 먼저 정한다

---

## 0. 이 문서를 먼저 쓰는 이유

이 프로젝트의 진짜 주제는 치매예방 앱이 아니라 **내 컴퓨터를 서버로 쓰는 것**이다.
앱은 소재이고, 위험은 **집 기기를 인터넷에 붙이는 행위 자체**에서 나온다.
그러므로 코드보다 위협 모델이 먼저다.

---

같은 물리 기기에 이미 살고 있는 것들. **앱보다 이쪽이 훨씬 비싸다.**

| 등급 | 자산 | 잃으면 |
|---|---|---|

| 🟠 상 | 회의실(~/meeting-room) — Node.js 주식 토론 시스템, 포트 3000 | 자체 구축 자산 + Vertex AI 과금 노출 |

| 🟡 중 | dementia-care 앱 데이터(인지 활동 기록) | 건강 관련 성격 |

> 핵심: **지켜야 할 1순위는 앱이 아니라 앱이 올라탈 기기다.**
> 앱은 침입 경로일 뿐이고, 값나가는 건 전부 그 옆에 있다.

## 2. 누가 온다고 가정하는가 (위협 행위자)

| 행위자 | 현실성 | 동기 | 대응 필요도 |
|---|---|---|---|
| **자동화 스캐너 · 봇넷** | ⭐⭐⭐⭐⭐ | 무차별. 노출 즉시 시작 | **최우선** |
| **나 자신의 설정 실수** | ⭐⭐⭐⭐⭐ | 없음(사고) | **최우선** |
| 기회주의적 공격자 | ⭐⭐⭐ | 채굴·봇넷 편입·랜섬 | 높음 |
| 공급망 오염(악성 패키지) | ⭐⭐ | 자격증명 탈취 | 중간 |
| 표적 공격자 | ⭐ | 없음(표적 가치 낮음) | 낮음 |

> **1위와 2위가 같은 별점**이라는 게 이 문서의 요지다.
> 밖에서 오는 놈만큼 내 손이 위험하다. 그래서 통제도 절반은 "내 실수를 무력화"에 쓴다.

## 4. 무엇으로 막는가 (통제)

### 원칙
> **주의력으로 막지 않는다. 경로를 없앤다.**
> "조심해서 설정한다" = 언젠가 뚫린다. "설정할 수가 없다" = 안 뚫린다.

### C-1. cloudflared를 격리 네트워크 안에 가둔다 ★ 최중요

| 방식 | 결과 |
|---|---|
| ❌ 호스트에서 실행 | `localhost` 전체가 사정권. 오타 하나로 5678 공개 |
| ✅ 격리 네트워크 안 컨테이너 | **`web:3000` 말고는 볼 수가 없다.** 겨누고 싶어도 경로 없음 |

→ AS-2를 **설정 문제에서 구조 문제로 바꾼다.** 실수해도 안전해진다.

### C-2. 노출 포트는 0개

- `api`, `db`: `ports:` 절대 사용 금지. `expose:`만 (= 도커 내부 전용)
- `web`: `ports:`도 불필요. cloudflared가 같은 네트워크에서 직접 부름
- **결과: 호스트의 어떤 포트도 새로 열리지 않는다**

### C-3. 전용 네트워크 (AS-4 차단)

```yaml
networks:
  dementia-net:
    driver: bridge
    internal: false   # cloudflared의 아웃바운드만 필요
```

- 기존 n8n·에이전트 네트워크에 **절대 attach 하지 않는다**
- `network_mode: host` **금지**
- 기본 bridge 네트워크 **사용 금지** (다른 컨테이너와 같은 방)

### C-4. 컨테이너 → 호스트 경계 (AS-3 차단)

| 항목 | 설정 | 이유 |
|---|---|---|
| Docker 소켓 마운트 | **절대 금지** | 소켓 = 호스트 root. 예외 없음 |
| `privileged: true` | **절대 금지** | 격리 해제 |
| 실행 사용자 | `user: "1000:1000"` (non-root) | 컨테이너 root ≠ 안전 |
| 권한 | `cap_drop: [ALL]` | 필요한 것만 다시 추가 |
| 권한 상승 | `security_opt: [no-new-privileges:true]` | setuid 차단 |
| 파일시스템 | `read_only: true` + 필요한 곳만 `tmpfs` | 침입자가 쓸 곳 없음 |
| 호스트 경로 바인드 | **금지**. named volume만 | `~` 마운트 = 개인 파일 노출 |
| 리소스 | `mem_limit`, `cpus` 설정 | 채굴·DoS 시 기기 마비 방지 |

### C-5. 비밀값 (AS-6 차단)

- `.env`는 **커밋 전에** `.gitignore`에 등록 (사후 등록은 이미 늦음)
- `.env.example`에 **키 이름만** (값 없이) 커밋
- Vercel식 `NEXT_PUBLIC_` 접두사 개념 = **브라우저 공개**. 비밀값에 절대 사용 금지
- DB 비밀번호: 랜덤 생성. 재사용 금지
- 커밋 전 점검: `git diff --cached` 눈으로 확인

### C-6. 공급망 (AS-5 완화)

- 베이스 이미지: 공식 이미지 + **태그 고정** (`python:3.12-slim`, `latest` 금지)
- `requirements.txt` 버전 핀 고정
- 런타임에 필요한 패키지만

### C-7. 앱 계층 (AS-8 차단)

- **정답 판정·난이도 계산은 전부 서버에서.** 프런트에 정답을 내려보내지 않는다
- 프런트 검증은 UX용. 공격자는 화면을 안 거치고 API를 직접 부른다
- 익명 UUID만. 이름·전화번호·생년월일 수집 안 함 (없는 데이터는 못 털린다)

### C-8. 접근 제어 (선택 — 권장)

Cloudflare Access를 앞에 세우면 공개 URL이 **초대된 사람만** 열 수 있게 된다.
개발 기간엔 이걸 켜두는 게 AS-1을 거의 0으로 만든다.
→ 무료 플랜 한도 및 자동 과금 여부는 **적용 전 별도 확인 필요**

## 5. 수용하는 위험 (Accepted Risks)

의식적으로 안 막기로 한 것들. 모르는 게 아니라 **알고 넘어간다.**

| 위험 | 왜 수용하나 |
|---|---|
| 서비스 중단 (데스크탑 꺼짐·재부팅) | 개인 프로젝트. 가용성 요구 없음. **이걸 위해 보안을 양보하지 않는다** |
| Cloudflare가 트래픽을 볼 수 있음 | 터널의 구조적 특성. 민감 데이터 자체를 안 다루므로 수용 |
| 0-day 취약점 | 개인 프로젝트 범위 밖 |
| 유동 IP·CGNAT | 터널이 우회하므로 무관 |

## 6. 배포 전 체크리스트

터널을 켜기 **전에** 전부 ✅ 여야 한다.

- [ ] `docker compose config` 출력에 `docker.sock` 문자열이 없다
- [ ] `docker compose config` 출력에 `privileged` 가 없다
- [ ] `ports:` 매핑이 **0개**다
- [ ] 호스트 경로 바인드 마운트가 **0개**다 (named volume만)
- [ ] 모든 서비스에 `user:` 가 non-root로 지정돼 있다
- [ ] `cap_drop: [ALL]` 이 모든 서비스에 있다
- [ ] `no-new-privileges:true` 가 모든 서비스에 있다
- [ ] 전용 네트워크를 쓰고, 기존 네트워크에 attach 하지 않는다
- [ ] `.gitignore` 에 `.env` 가 **첫 커밋 전에** 들어있다
- [ ] `git log -p` 로 비밀값이 이력에 없음을 확인했다
- [ ] 모든 이미지 태그가 고정돼 있다 (`latest` 없음)
- [ ] `mem_limit` 이 모든 서비스에 있다
- [ ] cloudflared가 **컨테이너로** 격리 네트워크 안에서 돈다 (호스트 아님)
- [ ] 터널 설정이 `web:3000` **하나만** 가리킨다
- [ ] 정답·난이도 로직이 서버에만 있다 (프런트에 정답 없음)
- [ ] 기존 n8n(5678)·에이전트·온담이에 **아무것도 안 건드렸다**

## 7. 사고 시 대응 (Runbook)

**이상 징후**: CPU 급등 / 모르는 아웃바운드 / 알 수 없는 컨테이너 / 계정 알림

**대응 순서**

1. `cloudflared` 컨테이너부터 정지 → **외부 경로 즉시 차단**
2. 나머지 컨테이너 정지 (삭제 금지 — 증거 보존)
3. Cloudflare 대시보드에서 터널 토큰 폐기
4. `docker logs` 로 진입 지점 확인
5. **호스트 침해가 의심되면**: 온담이·n8n·에이전트 credential **전부 교체**
6. 원인 규명 전 재가동 금지

> ★ 작업 원칙 적용: 오류·사고 상태에서 바로 재시도하지 않는다.
> 생성된 것 삭제 → 삭제 확인 → 재작업.
